# Hezalli ‚Äî Full Project Audit (2026-07-20)

Scope: architecture, backend correctness, database & performance, frontend quality,
testing & CI, documentation/configuration/deployment. **Security was explicitly
excluded** from this audit at the owner's request.

Method: six parallel deep-dive reviews over the full codebase, plus a verified
build/test baseline run in a clean environment. The highest-severity findings were
independently re-verified against the source before inclusion.

---

## 1. Executive summary

Hezalli is an unusually disciplined codebase for its size (~47k lines of hand-written
TypeScript): clean layering (RSC pages ‚Üí server actions ‚Üí `lib/` services), strict
client/server hygiene, exemplary bilingual/RTL support, a ledger-first wallet design
with a real reconciliation module, and a complete CI pipeline. Everything builds and
all tests pass.

The audit found **one systemic correctness defect** that accounts for most
high-severity findings: **status transitions that gate money movement check state
outside the transaction and update unconditionally inside it** (check-then-act /
TOCTOU). The codebase already uses the correct pattern ‚Äî
`updateMany({ where: { id, status: <expected> } })` + count check ‚Äî on all balance
*decrements* (stock, wallet spend, coupon maxUses, loyalty points), but not on state
*flips* (confirm top-up, mark withdrawal paid, complete/fail bill, cancel order,
apply refund, mark payout paid, settle sub-order). Applying that one pattern
consistently would eliminate the majority of high-severity items in one sweep.

The second theme is **unbounded queries on the storefront hot path**: the catalog
listing engine fetches every matching product and paginates in JavaScript, full-text
search has no GIN index, and several dashboards fetch all rows.

## 1a. Remediation ‚Äî round 1 (money-guard sweep)

A first follow-up change lands the systemic backend-correctness fix plus the
index/idempotency migration and real-action concurrency tests. Verified with
typecheck, lint, 79 unit + 106 integration tests (8 new), and a production build.

**Fixed in this round:**

- **TOCTOU status-flip guards** ‚Äî every money-gating flip now uses a conditional
  `updateMany({ where: { id, status: <expected> } })` + count check *inside* the
  transaction: `confirmTopUp` (¬ß3.1), `markWithdrawalPaid`/`rejectWithdrawal`
  (¬ß3.2), `completeBill`/`failBill` (¬ß3.3), `markPayoutPaid`/`rejectPayout`
  (¬ß3.10), `confirmPayment`/`rejectPayment` (¬ß3.11), `payPaymentRequest` (¬ß3.13).
- **Idempotency backed by DB constraints** ‚Äî partial unique indexes make
  `settleSubOrder` (¬ß3.8), loyalty `EARN` (¬ß3.8), and cashback (¬ß3.8) race-proof;
  the code now treats a unique violation as a no-op.
- **Reservation races** ‚Äî `transferEarningsToWallet` and `requestPayout` (¬ß3.4)
  compute the free balance and write under a `SELECT ‚Ä¶ FOR UPDATE` row lock on the
  seller balance, serialising concurrent outflows.
- **Non-atomic recompute** (¬ß3.12) ‚Äî `recomputeWalletBalance` and the seller
  `recomputeBalance` availableUsd now use a single `UPDATE ‚Ä¶ = (SELECT SUM‚Ä¶)`
  statement, closing the read-then-write window.
- **cancelOrder** (¬ß3.5, partial; ¬ß3.15) ‚Äî conditional double-cancel guard (no more
  double refund / double stock restore), stock/flip limited to still-cancellable
  sub-orders, and buyer cancel now restores redeemed loyalty points and frees the
  coupon usage (previously both were silently burned).
- **Seller `cancelSubOrder` refund** (¬ß3.6) ‚Äî cancelling a paid (wallet or
  admin-confirmed bank/USDT) sub-order now refunds the buyer to their HezalliPay
  wallet via `applyRefund`, marks it REFUNDED, restores redeemed loyalty points and
  stock, and notifies the buyer. COD / not-yet-paid orders still plain-cancel. The
  buyer no longer silently loses the money.
- **applyRefund** (¬ß3.7) ‚Äî cumulative refunds are capped to the paid amount under a
  sub-order row lock; over-refund is rejected.
- **expireStaleOrders** (¬ß3.9) ‚Äî no longer cancels orders whose payment proof is
  awaiting admin review; cancel is conditional on the order still being PENDING.
- **Indexes** (¬ß4.4) ‚Äî btree indexes added for `OrderItem.variantId`,
  `LedgerEntry.subOrderId`/`payoutId`, `LoyaltyTransaction.subOrderId`,
  `Payment.status`, `Payout.status`, `Order.createdAt`/`couponId`,
  `SubOrder.autoCompleteAt`, `WalletEntry.orderId`/`subOrderId`,
  `FlashSaleItem.variantId`, `Refund.subOrderId`, `Shipment.carrierId`,
  `User.referredById`, `Dispute.status`.

**Deliberately deferred (need their own PRs):** the remaining `cancelOrder`
state-machine work ‚Äî multi-store SHIPPED sub-order + already-refunded overlap
(¬ß3.5b‚Äìe); coupon `perUserLimit` atomicity (¬ß3.14); FTS GIN index (¬ß4.2,
needs the brand-join search query refactored to be index-usable); the storefront
listing SQL-pagination/caching work (¬ß4.1, ¬ß4.3, ¬ß4.8); a latent `getWalletId`
upsert race on a wallet's first-ever touch. All frontend, testing-breadth, and
docs findings below are unchanged by this round.

## 3. Backend correctness (money paths)

**The systemic defect:** balance *decrements* are guarded correctly everywhere
(conditional `updateMany` + `gte` + count check ‚Äî stock, flash stock, wallet spend,
coupon `maxUses`, loyalty points, withdrawal reservation). Status *flips* that gate
money movement are not. Each finding below was verified against the source; the two
most severe were independently re-checked line-by-line.

| # | Finding | Severity |
|---|---|---|
| 3.1 | **`confirmTopUp` double-credit** (`lib/actions/wallet-topup.ts:103‚Äì134`): `AWAITING_CONFIRMATION` check is outside the `$transaction`; inside, the update is unconditional. Two concurrent confirms (double-click, two tabs) both credit the wallet. Confirm+reject can also both succeed. *Verified.* | High |
| 3.2 | **Rejected withdrawal can be marked PAID** (`lib/actions/wallet-withdrawal.ts:131,181‚Äì203`): `markWithdrawalPaid` only blocks status `PAID`. A REJECTED withdrawal (funds already returned via ADJUSTMENT credit) can then be paid ‚Äî user gets both. Concurrent paid+reject and double-reject races too. | High |
| 3.3 | **Bill both delivered and refunded** (`lib/wallet-bills-core.ts:28,80‚Äì99`): `completeBill`/`failBill` read status then update unconditionally; provider auto-complete racing an admin "fail" gives the buyer airtime *and* a BILL_REFUND. | High |
| 3.4 | **`transferEarningsToWallet` mints money on double-submit** (`lib/actions/wallet-transfer.ts:35‚Äì72`): free-balance computed entirely before the tx; the tx writes unconditionally. Two concurrent "move all" submissions each pass, seller ledger goes negative, wallet credited 2√ó. `requestPayout` (`lib/actions/payout.ts:106‚Äì133`) has the same racy read-then-create. | High |
| 3.5 | **`cancelOrder` multi-defect** (`lib/actions/order.ts:500‚Äì593`): (a) concurrent double-cancel ‚Üí double refund + double stock restore; (b) refunds full `grandTotal` even when a sub-order was already refunded via `applyRefund`; (c) restores stock for sub-orders the seller already cancelled (inventory inflated); (d) aggregate-status guard lets a buyer cancel an order containing an already-SHIPPED sub-order; (e) flash-sale `soldCount` never decremented on cancel. | High |
| 3.6 | **Seller `cancelSubOrder` never refunds a paid buyer** (`lib/actions/seller-order.ts:52‚Äì121`): the `transition()` helper restores stock and flips statuses but moves no money. A wallet/bank-paid buyer whose sub-order the seller cancels loses the money ‚Äî no refund entry, no `Refund` record, no admin alert; the amount drops out of escrow accounting entirely. *Verified.* | High |
| 3.7 | **`applyRefund` allows unlimited cumulative partial refunds** (`lib/refunds.ts:61‚Äì95`): prior refunds are never summed ‚Äî three $50 partials on a $60 sub-order all succeed. Also a concurrent full-refund double-credit race (e.g. seller double-clicks "confirm return received"; `return.ts:280‚Äì335` flips ReturnRequest status before calling `applyRefund`). | High |
| 3.8 | **Settlement double-credit** (`lib/finance.ts:151‚Äì201`): `settleSubOrder` is non-transactional; idempotency is a `count()` check with no unique constraint on `LedgerEntry(subOrderId, type)`. It's called concurrently from buyer `confirmReceived`, cron, *and* lazily on every buyer orders-page load ‚Äî both writers can pass `count === 0` and double-credit the seller. Same racy `findFirst` idempotency for loyalty EARN (`lib/loyalty.ts:31`) and cashback (`lib/wallet-cashback.ts:42`). Also `LedgerEntry.subOrderId` is unindexed, so the check is a table scan. | High |
| 3.9 | **`expireStaleOrders` cancels orders with payment proof under review** (`lib/actions/payment.ts:237‚Äì252`): filters only `Order.status = PENDING` ‚Äî submitting proof keeps the order PENDING (only `payment.status` changes). A buyer who paid at hour 23 gets auto-cancelled at hour 24+ while admin review is pending. Runs from cron *and* every `/account/orders` page load. | Med-High |
| 3.10 | **`markPayoutPaid`**: double-pay race + REJECTED‚ÜíPAID allowed (`lib/actions/payout.ts:159‚Äì190`). | High |
| 3.11 | `confirmPayment` can confirm an expired/cancelled order (`lib/actions/payment.ts:105‚Äì124`) ‚Äî order becomes CONFIRMED while all sub-orders stay CANCELLED and stock was resold; a state no other code handles. | Medium |
| 3.12 | Balance recompute is not atomic (`lib/wallet.ts:31‚Äì41`, `lib/finance.ts:76‚Äì118`): aggregate then update as two statements outside any lock ‚Äî a stale aggregate can overwrite and resurrect concurrently-spent funds. Use a single atomic `UPDATE ... SET x = (SELECT SUM...)` or a row lock. Related: recomputes run *after* the writing tx commits (`order.ts:449‚Äì469`), so a crash in between leaves `Wallet.availableUsd ‚âÝ Œ£ entries` until the next recompute. | Medium |
| 3.13 | `payPaymentRequest` marks the request PAID only *after* the transfer (`lib/actions/wallet-request.ts:68‚Äì89`) ‚Äî two payers can both fund the same request. | Medium |
| 3.14 | Coupon `perUserLimit` not enforced atomically (`lib/vouchers.ts:134‚Äì141`; unique is `[couponId, orderId]`, not per-user) ‚Äî N parallel checkouts redeem a limit-1 coupon N times. | Medium |
| 3.15 | `cancelOrder`/`expireStaleOrders`/seller cancel never restore redeemed loyalty points or decrement `coupon.usedCount` ‚Äî points burned, limited coupons consumed by cancelled orders (contrast `applyRefund`, which restores points proportionally). | Medium |
| 3.16 | Velocity (AML) limits checked outside the debit tx (`lib/wallet-velocity.ts:48‚Äì79`) ‚Äî parallel sends can exceed daily/monthly caps; refunded failed bills still consume outflow headroom (BILL_REFUND not netted). | Low |
| 3.17 | Wallet PIN lockout counter is read-modify-write (`lib/wallet-pin.ts:62‚Äì73`) ‚Äî parallel wrong-PIN attempts exceed `MAX_ATTEMPTS`; use atomic `increment`. | Low |
| 3.18 | Line-total rounding mismatch (`lib/actions/order.ts:131,207,319`): per-store `itemsTotal` sums unrounded floats while stored `lineTotal`s are rounded independently ‚Äî cent-level drift propagates into commission/refund math. Round lines first, sum the rounded values. | Low |
| 3.19 | Return window never closes when both `completedAt` and `shipment.deliveredAt` are missing (`lib/actions/return.ts:86‚Äì89`) ‚Äî fail closed instead. | Low |

**Done well:** ledger-first design with immutable entries + reconciliation module
(`lib/wallet-reconcile.ts`); textbook conditional-decrement guards on all hot debit
paths; withdrawal reservation at request time; `Decimal(12,2)` in Postgres with
consistent `round2`; proportional voucher/points allocation with
last-group-absorbs-rounding; `applyRefund`'s proportional seller-ledger reversal;
server-side re-derivation of all prices.
